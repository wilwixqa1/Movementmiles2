WHAT WAS BUILT (all live on main, Railway auto-deployed)
=====================================================================

A. ENGAGEMENT SYNC ENGINE
   Tables:
   - ymove_engagement: one row per member per sync run. Full raw JSONB payload
     kept + extracted cols (programs_total/active, active_program_ids/titles,
     all_program_titles, workout_count, last_workout_at, program_sessions_count,
     ymove_subscription_active). DESIGN RULE: insights read ONLY the latest row
     per member; old snapshots stored but never aggregated (Will's noise call).
   - engagement_sync_runs: run log (status, progress, results, errors).
   Function run_engagement_sync(limit, batch_label, scope):
     scope='all'    -> every member ever incl canceled (one-time backfill)
     scope='active' -> status active/trialing only (weekly re-sync; canceled
                       members' history is frozen so re-fetch is pointless)
   Batched: 20 concurrent lookups/batch, ~1s between batches, one 429 retry
   each. (Measured: 40 members 85.9s sequential vs 18.9s batched.)
   Endpoints:
     POST /api/admin/engagement-sync-now  {confirm|dry_run, limit, scope}
          dry_run makes ZERO ymove calls.
     GET  /api/admin/engagement-sync-runs
     GET  /api/admin/engagement-sample?limit=N
   Weekly schedule: Mon 3:00 AM ET, scope=active. GATED behind
   ENGAGEMENT_SYNC_ENABLED=1 (see open action 1).

B. HISTORICAL BACKFILL (DONE -- run id 2, batch s36_engagement_all_20260716_0521)
   7,929 processed / 7,245 found / 684 not_found / 0 errors / ~4h39m.
   684 not_found = old imports ymove has no record of (~8.6%).

C. RETENTION BY PROGRAM card (below Channel Performance)
   GET /api/admin/program-insights. Latest engagement snapshot per member
   JOIN latest subscription per member. Outcome classes:
     trialing / paid_active (active+trial past, IGNORES converted_at to dodge
     the gap) / paid_canceled (converted_at set) / trial_canceled.
   Conv Rate = paid/(paid+trial_canceled) resolved-only. Shared attribution
   (member counts in each active program). "(No Program)" = comparison cohort.
   Rows <20 followers dimmed. Subtitle shows live member count + sync date.

D. ACTIVATION BY WORKOUT COUNT card (below Retention by Program)
   GET /api/admin/activation-insights. Buckets (0,1-2,3-5,6-9,10-14,15-24,
   25-49,50+) across SIGNUP-ANCHORED windows: First 30d / 60d / 90d / Lifetime.
   Signup-anchored (not calendar) so churned + active members are comparable.
   Per window: members, conv rate, paid cancel rate, still-active %, avg tenure.
   Conv-rate bars: ABSOLUTE 0-100% scale + orange baseline tick at the window's
   overall rate (find the inflection = where bars cross the tick).
   Second table: workouts-at-cancellation distribution (trial vs paid).
   Window toggle is client-side (all windows in one API response).

E. MEMBER_WORKOUTS ACCUMULATOR (the North Star infrastructure)
   Table member_workouts: UNIQUE(entry_id) so duplicate ingestion is a no-op.
   The 50-cap limits how far BACK we see, not recency; weekly overlapping
   snapshots + dedupe-by-entry-id reconstruct COMPLETE histories over time.
   - Sync auto-ingests each snapshot's workouts (helper _s36_ingest_workouts).
   - POST /api/admin/seed-member-workouts {confirm:true} -- one-time
     retroactive seed from all stored snapshots. ALREADY RUN: 139,380 unique
     workouts across 6,134 members; a re-run added 0 (dedupe proven).
   COLD STORAGE: nothing on the dashboard reads this yet. It accrues weekly
   and becomes usable in ~3-6 months for the intensity table (see North Star).

KEY FINDINGS (Jul 16 data -- share-worthy)
=====================================================================
- Program followers convert 52-58%; NO-program members convert 42.5% and
  average ~6.9 workouts vs 27+ for followers. (Correlation, not causation.)
- ACTIVATION THRESHOLD (First 30d): conv climbs to ~53.9% at 10-14 workouts
  then FLATTENS (54.4% at 15-24). Gains start at 3-5. 1-2 workouts is no
  better than 0. => target = ~10-14 workouts in the first month.
- Lifetime view conv climbs forever (27%->77%) = TENURE ARTIFACT (staying
  longer manufactures workouts). Kept as a cautionary contrast only.
- CHURN IS BIMODAL: a big low-engagement mass (0-5 workouts, mostly trial
  cancels = activation problem) AND a heavy-user mass (50+ workouts, ~508
  PAID cancels = fatigue/completion problem). Two problems, two fixes.

NORTH STAR METRIC (Ahmed's real ask -- infra built, data accruing)
=====================================================================
GOAL: workouts-per-week bucket -> avg realized LTV + avg retention time.
"How many workouts/week makes a stable, high-value member?"
- CANNOT be done retrospectively: the 50-cap censors exactly the long-tenure
  high-intensity members (their early workouts scrolled out). Their cells
  would be empty/fake.
- CAN be done prospectively: the member_workouts accumulator (E) captures
  complete histories from Jul 2026 forward. Build the bucket table on top in
  ~3-6 months once depth accrues. LTV = renewals x plan price ("realized
  revenue", not projected). Retention = tenure, both from subscriptions.
- Dedupe = counting each workout once via UNIQUE(entry_id); needed because
  weekly snapshots overlap ~47/50. Runs automatically now.

  a health check; use /api/admin/stats.

=====================================================================
STRATEGY DISCUSSION (context, not a task)
=====================================================================
Explored generalizing M&M into a product. Conclusions:
- Horizontal "Stripe/Apple/Google analytics" = saturated (Baremetrics,
  ChartMogul, Braavo, RevenueCat...). Baremetrics already does cross-platform;
  it wins on financial NORMALIZATION but DROPS per-member/email/trial detail
  (esp. Google Play) -- exactly the granular layer M&M is built on.
- ymove-vertical is the tighter idea: same pipeline for every customer, so our
  scar tissue transfers. Onboarding simple (Apple/Google come THROUGH ymove,
  no App Store Connect dance). BUT: Tosh/ymove becomes a hard dependency under
  100% of tenants; TAM likely small (ymove $39/mo indie platform; Meg is a
  top-decile creator, not typical); the code is not a moat (rebuildable).
- Multi-tenant would still run on Railway (one deployment, tenant_id column,
  encrypted keys in DB / OAuth via Stripe Connect -- NOT env vars per customer).
- Will's lean: partner with Tosh (he seems uninterested in high-tier analytics
  because most of his creators don't need it). NEXT STEP before any build = an
  exploratory Tosh conversation (distribution/referrals, how many Meg-sized
  creators exist, blessed multi-tenant API access, native attribution keys)
  AND a Meg conversation first (the dashboard is her paid deliverable; stake/
  design-partner/reference question).
- Comparative take: SBP is likely the stronger business to build (no platform
  dependency, real accuracy moat, clear buyer, binary validation milestone).
  Keep M&M as a paying maintenance client; don't turn it into a startup unless
  the Tosh conversation opens a real channel.
