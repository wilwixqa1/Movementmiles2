1. ENGAGEMENT SYNC (backend)
   - Table ymove_engagement: one row per member per sync run. raw JSONB keeps
     the full ymove user payload; extracted columns for querying (programs
     total/active, active_program_ids/titles, workout_count, last_workout_at,
     program_sessions_count, ymove_subscription_active).
   - Design rule: insights read ONLY the latest row per member (Will's call:
     older snapshots would over-represent abandoned programs; raw history is
     stored but never aggregated).
   - Table engagement_sync_runs: run log w/ progress, results, errors.
   - run_engagement_sync(limit, batch_label, scope):
       scope='all'    -> every member ever (one-time backfill, DONE, never again)
       scope='active' -> active/trialing only (weekly; canceled members'
                         history is frozen so re-fetching is pointless)
   - Batched: 20 concurrent lookups per batch, ~1s between batches, one 429
     retry per request (Tosh OK'd Jul 17). Was 1 req/sec sequential.
   - Endpoints:
       POST /api/admin/engagement-sync-now  {confirm|dry_run, limit, scope}
            dry_run makes ZERO ymove calls.
       GET  /api/admin/engagement-sync-runs
       GET  /api/admin/engagement-sample?limit=N
   - Weekly schedule: Mon 3:00 AM ET, scope=active (~2,006 members, ~4 min
     batched). GATED behind Railway env ENGAGEMENT_SYNC_ENABLED=1.
     STATUS AT HANDOFF: flag not yet set -- Will to arm it.

3. RETENTION BY PROGRAM (dashboard card, below Channel Performance)
   - GET /api/admin/program-insights. Latest engagement row per member JOIN
     latest subscription per member (email lowercased both sides).
   - Outcome classes: trialing / paid_active (active + trial past, deliberately
     ignores converted_at) / paid_canceled (converted_at set) / trial_canceled.
   - Conv Rate = paid/(paid+trial_canceled) resolved-only; Paid Cancel Rate =
     paid_canceled/paid. Shared attribution across a member's active programs.
     "(No Program)" = ymove-found members following nothing (comparison cohort).
   - Rows <20 followers dimmed w/ warning glyph. Overall baseline row pinned.
   - Subtitle shows live member count + last sync date from the API.

== LANDMINES LEARNED ==
   - ROUTE ORDER: @app.get("/{path:path}") catch-all at ~line 14200 serves the
     marketing site. Any GET endpoint defined AFTER it in main.py is silently
     swallowed (200 + HTML). POSTs unaffected. Register new GETs BEFORE it.
   - Deploy verification must check RESPONSE BODY, not status code (catch-all
     returns 200 for everything).
   - NO GIT PUSH WHILE A SYNC IS RUNNING: Railway redeploys on push and kills
     the background task. Corollary: a push IS the emergency stop for a
     runaway sync (no cancel endpoint exists).
   - inspect-ymove-user queries ymove for any non-empty email -- don't use it
     as a health check. Use /api/admin/stats.
