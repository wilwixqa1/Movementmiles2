APPROACH A vs APPROACH B -- CHANNEL PERF DESIGN DECISION
=========================================================
The channel performance table currently uses Approach A (event-based). This is a deliberate
trade-off that was discussed extensively.

APPROACH A (CURRENT): Count events by when they happened
  - Trial Starts: subs created in window
  - Still Trialing: subs created in window whose trial is still running
  - Trial Canceled: cancellations that HAPPENED in window (any creation date)
  - Paid Conv: conversions that HAPPENED in window (any creation date)
  - PRO: Cancellation/conversion numbers match dashboard churn exactly
  - CON: Columns don't add up (starts ≠ trialing + canceled + converted)
         because cancellations include older subs

APPROACH B (COHORT): Count everything by when the person signed up
  - Trial Starts: subs created in window
  - Still Trialing: subs created in window, still in trial
  - Trial Canceled: subs created in window that are now canceled
  - Paid Conv: subs created in window that converted
  - PRO: Numbers always add up (starts = trialing + canceled + converted)
  - CON: With 30-day trials and 30-day window, almost everyone is "still trialing"
         Cancellation/conversion numbers look artificially low
         Numbers don't match dashboard churn

The "Trial-to-Paid Conversion (Rolling Windows)" table elsewhere on the dashboard already
implements Approach B correctly. The channel perf table uses Approach A to match the
dashboard totals. Both views exist and serve different purposes.

SQUARESPACE UTM TRACKING SCRIPT
=================================
Location: squarespace_utm_script.html in repo root
Updated: May 16, 2026 (S34)
Status: DEPLOYED by Ahmed (utm_content and utm_term are flowing to page_views)

What it does:
1. Captures UTM params from URL, stores as cookies (30-day expiry, mm_ prefix)
2. Auto-sets mm_lp cookie with current pathname (landing page)
3. Rewrites ymove checkout links to include all stored UTM cookies as URL params
4. Posts page-view to backend with UTM params

Cookie list: utm_source, utm_medium, utm_campaign, utm_term, utm_content, ym_source, mm_lp
Page-view POST includes: utm_source, utm_medium, utm_campaign, utm_content, utm_term
(does NOT send mm_lp or ym_source to page-view endpoint)

Note: mm_lp is captured and appended to ymove links, but ymove doesn't store it (see above).


DATA FLOW: UTM → SUBSCRIPTION
===============================
1. User clicks UTM link → lands on Squarespace site
2. squarespace_utm_script.html captures UTMs from URL, stores in cookies
3. Script sets mm_lp cookie to current pathname
4. When user clicks ymove checkout link, script appends all cookies to URL
5. User subscribes through ymove → Stripe subscription created
6. Stripe fires customer.subscription.created webhook → main.py stores sub
7. _ymove_lookup_utm() queries ymove Member Lookup API, reads meta field
8. _store_utm_on_subscription() writes UTM fields + landing_page to sub record
9. For Apple/Google: ymove fires webhooks → main.py stores sub (no UTMs, app store strips them)

When trial-to-paid conversion happens:
- Stripe sends customer.subscription.updated webhook
- previous_attributes.status = 'trialing', new status = 'active'
- We stamp converted_at = NOW()
- This works for Stripe subs going forward (0 missing in last 30d)
- Apple/Google subs don't get this webhook, so converted_at must be backfilled
