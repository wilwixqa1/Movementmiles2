WHAT WAS BUILT (all live on main)
=====================================================================

A. SCHEMA (commit 5fccffa, main.py startup Block 3c ~line 373)
   Three nullable columns on subscriptions, added via the existing idempotent
   ADD COLUMN IF NOT EXISTS pattern:
   - cancellation_feedback  TEXT -- the CUSTOMER's selection from the portal
     (too_expensive, unused, switched_service, missing_features, low_quality,
      customer_service, too_complex, other)
   - cancellation_comment   TEXT -- optional free text the customer typed
   - cancellation_reason    TEXT -- STRIPE's own classification, NOT the
     customer's (cancellation_requested, payment_failed, payment_disputed)
   CRITICAL DISTINCTION: feedback != reason. `reason` is present on nearly every
   Stripe cancel going back years; `feedback` only exists for self-serve portal
   cancels. Conflating them produces a wildly inflated coverage number (see
   MISTAKES MADE).
   Purely additive. No existing column, value, or query was modified anywhere
   in S37. Total diff across the session: +717 / -34 lines.

B. WEBHOOK CAPTURE (commit 5fccffa, main.py ~line 995)
   Separate UPDATE after the main subscription upsert, deliberately NOT folded
   into it, so a failure here can never break subscription ingestion.
   - Fires on BOTH customer.subscription.updated AND .deleted. This matters:
     feedback is set the moment the customer clicks cancel in the portal, which
     under the S26 period-end semantics is an .updated with
     cancel_at_period_end=true. .deleted may not arrive for weeks. Capturing
     only on .deleted would lose everyone still in their grace period.
   - COALESCE on the write: a captured reason is never overwritten with a later
     NULL from an unrelated .updated event.
   - Known behavior, accepted by Will: if a customer cancels (reason recorded)
     then reverses the cancel while still in grace, the reason STAYS on the
     record. Judged interesting rather than wrong. Flag if this ever distorts a
     count.
   THIS IS NOW THE SYSTEM OF RECORD. See LANDMINES re: Stripe retention.

C. BACKFILL, v1 -- BLOCKING (commit 5fccffa)
   POST /api/admin/backfill-cancellation-reasons {preview|confirm, limit}
   Simple synchronous loop. Works, but blocks the event loop. Fine at limit=500,
   would have frozen the whole app for ~45 min at full scope. Superseded by v2.
   Left in place because its {preview: true} mode is still the fastest way to
   sample what Stripe holds for the first 5 unfilled rows.

D. BACKFILL, v2 -- BACKGROUND (commit 231c06e)
   run_cancellation_reason_sync(limit, days, batch_label)
   - asyncio.to_thread around the synchronous stripe library call. The codebase
     had NO existing thread-offload pattern; a tight loop of thousands of
     stripe.Subscription.retrieve() calls blocks the event loop and takes the
     entire dashboard down for the duration, for every user, not just the one
     who started it.
   - days=0 -> all history (the one-time catch-up). days=N -> last N days.
   - Progress logged to the EXISTING engagement_sync_runs table with an
     s37_cxl_ batch prefix. No new table; poll the same endpoint as S36.
   - Writes ONLY the three S37 columns plus import_batch/updated_at, and every
     run emits its own revert_sql in the results JSON.
   POST /api/admin/cancellation-reason-sync-now {confirm, limit, days}

E. WEEKLY RECONCILIATION (commit 231c06e, scheduler ~line 450)
   run_weekly_cancellation_reason_sync -- Mondays 4:00 AM ET, one hour after
   the S36 engagement sync so the two never overlap.
   SCOPE IS DELIBERATELY NARROW: last 30 days only. The webhook is the primary
   capture path and handles new cancels in real time. This job exists ONLY to
   catch events missed during a deploy or outage. A full-history rescan every
   week would be thousands of Stripe calls to re-confirm rows already known to
   be empty.
   Gated on STRIPE_SECRET_KEY being present. No new env var needed.

F. GET /api/admin/churn-reasons (commit 5fccffa, main.py ~line 14203)
   REGISTERED BEFORE THE CATCH-ALL. Returns per-reason counts with TWO
   percentages (see KEY FINDINGS), a by_month breakdown, coverage_pct, and
   comment_total. Accepts ?months=N. require_viewer, so the read-only viewer
   password works.
   Bucketing logic: feedback if present -> else involuntary_* if Stripe's reason
   is payment_failed/payment_disputed -> else no_reason_recorded.

G. GET /api/admin/churn-comments (commit 9c6beef)
   Paginated free text. ?months=N&reason=X&page=N&per_page=25&format=csv
   - Separate endpoint so the summary card does not drag comment rows around on
     every dashboard refresh.
   - Returns facets (which reasons actually have comments, with counts) to
     populate the filter dropdown.
   - format=csv returns ALL matching rows, ignoring pagination, honoring the
     active window and reason filter. This is the backup-off-Stripe mechanism.

H. "WHY MEMBERS CANCEL" CARD (static/admin.html, below Activation by Workout)
   - Window toggle: 3 / 6 / 12 Mo / All Time (default All Time)
   - Table: reason, members, % of all churn, % of those who told us, bar
   - "No reason recorded" row is DIMMED with a grey bar. It is the largest row
     and would otherwise visually dominate. Dimming keeps it honest without
     making it the headline.
   - Comments: 25 per page, prev/next, reason filter with counts, Export CSV
   - Window change resets pagination to page 1 (otherwise switching to a
     narrower window lands on a page that no longer exists)
   - loadChurnReasons() wired into the main dashboard refresh alongside the S36
     loaders (~line 1428)

LANDMINES / RULES LEARNED THIS SESSION
=====================================================================
- STRIPE APPEARS TO EXPIRE cancellation_details. Will observed the earliest
  written comment was 4/30, and that YESTERDAY the earliest was 4/29. A
  boundary that advances one day per day is a ROLLING RETENTION WINDOW, not a
  feature-rollout date. Implication: roughly 85-90 days of history is all that
  was ever recoverable, the backfill captured the maximum available, and THE
  WEBHOOK IS NOW THE ONLY THING PRESERVING THIS DATA. NOT YET CONFIRMED against
  Stripe docs -- confirm in S38. If the real window is under 30 days the weekly
  reconciliation is useless and needs rethinking. Consider widening it to 60
  days regardless; the cost is API calls, the benefit is margin for error.
- SILENT FAILURE IS THE REAL RISK. If the webhook stops capturing, the card
  just shows fewer reasons and still looks plausible. Nobody would notice until
  the data had already aged out of Stripe permanently. A staleness check
  (alert if no new reason captured in 7 days, surfaced on the dashboard or in
  the daily digest) was discussed and NOT built. Recommended for S38.
- THE STRIPE LIBRARY IS SYNCHRONOUS. Any loop over more than a few hundred
  stripe calls must use asyncio.to_thread or it blocks the event loop and takes
  the whole app down. This codebase had no prior example of this pattern.
- ROUTE ORDER (S36 rule, still in force): @app.get("/{path:path}") at ~line
  14729 serves the marketing site. ANY GET registered after it is silently
  swallowed and returns 200 + HTML. Both S37 GET endpoints are registered in
  the marked safe zone above it. POSTs are unaffected.
- ADMIN AUTH IS A HEADER, NOT A QUERY PARAM. X-Admin-Password. Burned several
  messages on this. ADMIN_PASSWORD and VIEWER_PASSWORD are read from env with
  hardcoded fallbacks at main.py lines 40-41; neither env var is set on
  Railway, so the fallbacks are live. Will has assessed and accepted this. DO
  NOT COMMENT ON IT.
- NO GIT PUSH WHILE A SYNC RUNS. Railway redeploys on push and kills background
  tasks. A push is also the emergency stop (there is no cancel endpoint).
- ENGAGEMENT SYNC ESTIMATE IS ~4.5x OPTIMISTIC. The dry run predicted 4.1 min
  for 1,976 members; actual was 18.4 min. The estimator uses a flat
  0.125s/member; real throughput was ~0.56s. Harmless at 3 AM but the dry run
  will mislead on any larger scope. Worth correcting.
- Standing rules still in force: read before edit, one push at a time with
  Will's visual confirmation, dry-run/preview before data mods, batch-tag all
  data mods, NO EM DASHES in outbound prose, anti-anchoring (2 pushbacks ->
  re-examine).
