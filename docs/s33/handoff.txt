ARCHITECTURE NOTES FOR NEXT SESSION
=====================================

DATA FLOW: UTM -> SUBSCRIBER
  1. User clicks UTM link -> lands on Squarespace site
  2. squarespace_utm_script.html captures UTMs from URL, stores in sessionStorage
  3. When user clicks ymove checkout link, script appends stored UTMs to the ymove URL
  4. User subscribes through ymove -> Stripe subscription created
  5. Stripe fires customer.subscription.created webhook -> main.py stores sub with UTMs
  6. For trials: Stripe fires customer.subscription.updated when trialing->active -> main.py stamps converted_at
  7. For Apple/Google: ymove fires webhooks -> main.py stores sub (but no UTMs since app store strips them)

CONVERSION TRACKING
  - Stripe: converted_at stamped on trialing->active transition (customer.subscription.updated webhook, line ~980)
  - Apple/Google: converted_at stamped via smart-backfill-conversions endpoint (line ~2184)
  - Trial period is ~1 month free trial
  - Current 90-day channel perf window means cohorts from 60+ days ago should show conversion data
  - Subs created in last 30 days are likely still trialing, which is expected

AUTH SYSTEM
  - Simple password-based, no tokens or sessions
  - Frontend stores password in JS var (adminPw), sends as X-Admin-Password header on every request
  - No JWT, no cookies, no session management
  - Future: Will mentioned possibly adding Google SSO. Would need a real auth system with JWT tokens.

DATABASE TABLES (key ones)
  - subscriptions: main table. Has utm_source, utm_medium, utm_campaign, utm_term, utm_content, source (apple/google/stripe), status, converted_at, effective_canceled_at, readable_id
  - page_views: utm_source, utm_medium, utm_campaign, utm_content (added S33)
  - leads: signup form submissions with UTM data
  - utm_links: saved UTM link configurations (label, base_url, all UTM fields, full_url)
  - ad_spend: monthly spend by channel for CPA calculations
  - reconciliation_snapshots: daily snapshot data for monitoring

STATS ENDPOINT (/api/admin/stats)
  - Single massive endpoint that returns ALL dashboard data
  - ~700 lines of queries
  - Returns: subscriber counts, conversion counts, churn, MRR, channel_performance, conversion_funnel, trend_weekly, trend_monthly, untagged_by_source, leads data, AI insights
  - This is the bottleneck endpoint -- every dashboard load hits it

FRONTEND (admin.html)
  - Single-file vanilla JS dashboard, no framework
  - All charts are hand-built SVG (no Chart.js or D3)
  - Quick-btn class used for toggle buttons -- needs both .date-bar scoped and standalone CSS
  - renderChannelPerfTable(), renderUtmTrendChart(), renderConvFunnel() are the main chart functions
  - Window globals: _channelPerf, _utmTrendWeekly, _utmTrendMonthly, _untaggedBySource, etc.
